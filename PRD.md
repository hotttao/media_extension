5. 等待任务完成
6. 保存生成的模特图到数据存储中

## 微服务重构
我现在想把 local_bridge 这个服务重构为一个微服务，可以通过 api 请求接受任务。同样也支持当前通过命令行提交任务的方式。所有任务都存储到数据库中。

task 保存在 media_ai_task 表中。表中需要包含如下字段:
透传字段: 这些字段都是上游服务通过参数传递过来的。没有传递默认为空字符串。
1. user_id: 用户ID
2. team_id: 团队ID
3. ip_id: 虚拟IP ID
4. product_id: 产品ID

任务详情字段:
2. task_type: 任务类型，例如: 图生视频、图生图、定妆图、首帧图
3. params: 任务参数 json 字符串，保存了每个任务的详细参数。输入如果是图片，保存的就是图片的 url
4. prompt: 任务提示词
5. output: 输出图片或者视频的保存路径，请求参数中包含使用请求参数，否则内置规则生成。
6. status: 任务状态
7. error: 任务错误信息
6. start_at: 任务开始时间
8. complete_at: 任务完成时间
8. create_at: 创建时间


**你需要使用 DDD 的设计模式和思想完成任务重构**

重构后，代码结构清晰。每个领域都需要使用类似 Pydantic Model 类似的元数据的方式，严格定义输入和输出。

在重构之前，你需要重新规划整体的目录机构，保证目录结构清晰。
重构完成后，原本支持的命令行提交任务的方式，需要重新实现。
